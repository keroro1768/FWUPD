# FWUPD 任務規則

> Heartbeat 每 30 分鐘自動讀取此檔案，確保團隊遵循統一流程。
> 最後更新：2026-03-30

---

## 1️⃣ 任務狀態

| 標記 | 意義 |
|------|------|
| ⏳ 待處理 | 等待執行 |
| 🔄 進行中 | 正在執行 |
| ✅ 完成 | 已完成 |
| ⏸️ 暫停 | 硬體/外部 Pending |
| ⏰ 延後 | 暫時擱置 |

---

## 2️⃣ 執行流程

```
定時觸發（每 30 分鐘）
│
▼
① 任務檢查
│
├── 有 🔄 → 回報進度
│
├── 有 ⏳ → 自動執行
│ │
│ ▼
│ 完成後 → 銜接下一個 ⏳
│ │
├── 無 ⏳ → Review & 新增任務
│ │
│ ▼
└── 自動新增的任務 → 立即指派並執行
```

---

## 3️⃣ Review 自動新增邏輯

| 條件 | 自動新增並執行 | 指派 |
|------|----------------|------|
| 硬體 Pending | Mock/Stub + 驗收標準 | Giroro |
| 缺少文件 | 文件任務 | Dororo |
| 缺少測試 | 測試任務 | Dororo |
| 有優化空間、需要分析、調查、蒐集資料、情報分析 | 優化、增強功能任務 | Kururu |

**優化職責：**
- 不僅就當前任務檢查是否經得起測試
- 還要主動提出新的可強化、增加的功能
- 提供建議交 Keroro 決策後執行

| 成員 | 職責 |
|------|------|
| **Keroro** | Commander（統籌協調）|

## 5️⃣ 任務資料夾結構

```
tasks/
├── TaskList.md         ← 專案任務總覽（Heartbeat 讀取）
├── TaskRules.md        ← 本檔案（任務規則）
├── Task_Template.md    ← 新任務模板
├── Verify_Template.md  ← 驗收模板
├── F001/               ← 任務編號（F = FWUPD）
│   ├── Task.md         ← 任務描述
│   ├── PLAN.md         ← 執行計畫
│   ├── WORKLOG.md      ← 工作日誌
│   └── VERIFY.md       ← 驗收結果
└── enhancement/
    └── E001_xxx/       ← Enhancement 提案
```

---

## 6️⃣ Enhancement 提案規則

### 提案時機

- 任何時候想到「這裡可以更好」
- Review 會議時提出
- 不需要等到有時間，提案只是構想

### 提案格式

```
tasks/enhancement/E001_xxx/
├── Task.md       ← 強化構想
├── PLAN.md      ← 執行計畫
├── WORKLOG.md   ← 工作日誌
└── VERIFY.md   ← 完成後填寫
```

FWUPD 專案文件存放在以下位置：

| 類型 | 路徑 |
|------|------|
| 知識庫 | `knowledge/` |
| 流程文件 | `doc/` |
| 任務文件 | `tasks/` |
| Brainstorm | `Brainstorm/` |

## 8️⃣ 研究 vs 實作 任務區分

| 類型 | 標記 | 說明 |
|------|------|------|
| 研究 | 🔬 | 收集資料、分析、評估 |
| 實作 | 🔧 | 開發、建置、部署 |
| 驗證 | 🔍 | 確認正確性、完整性 |
